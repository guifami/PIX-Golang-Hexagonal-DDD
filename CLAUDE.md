## Commands

```bash
# Full stack (Postgres + Kafka + Migrations + App)
make up
make down
make rebuild        # destroys all volumes, rebuilds from scratch

# Local run (requires .env.local with DB_HOST=localhost, KAFKA_BROKER=localhost:9092)
make run

# Regenerate Swagger docs (required after changing handler annotations)
make swag

# Format and tidy
make fmt
make tidy
```

Go tests (none yet — run when added):
```bash
go test ./...
go test ./internal/application/usecase/...
```

## Environment

- `.env` → used inside Docker (DB_HOST=go_db, KAFKA_BROKER=kafka:9092)
- `.env.local` → used for `make run` (DB_HOST=localhost, KAFKA_BROKER=localhost:9092)
- Migrations run automatically via the `migrate` Docker Compose service on `make up`
- Kafka UI available at `http://localhost:8080` when running via Docker

## Architecture

Hexagonal architecture (ports & adapters) + DDD. Dependency flow is strictly inward:

```
HTTP Handler / Kafka Consumer → Port In (interface) → UseCase → Port Out (interface) → Postgres / Kafka Producer
```

### Domain Layer (`internal/domain/`)

Pure Go — zero external dependencies. Three aggregates:

- **`account/`** — `Account` entity with `Debit`/`Credit`/`Activate`/`Deactivate` methods. Value objects: `CPF` (validates check digits), `Money` (int64 cents, never float), `AccountStatus`.
- **`pixkey/`** — `PixKey` entity. Value object `KeyType` (CPF/CNPJ/PHONE/EMAIL/EVP) with regex validation per type.
- **`transaction/`** — `PixTransaction` entity with state machine: `PENDING → PROCESSING → COMPLETED | FAILED`. Methods: `StartProcessing()`, `Complete()`, `Fail()`.

All domain errors are sentinel vars (`errors.Is` compatible). Adapters map them to HTTP/SQL-specific representations.

### Application Layer (`internal/application/`)

**Ports in** (`ports/in/`): `AccountUseCase`, `PixKeyUseCase`, `PaymentUseCase` — interfaces the HTTP handlers depend on.

**Ports out** (`ports/out/`): `AccountRepository`, `PixKeyRepository`, `TransactionRepository`, `EventPublisher` — interfaces the use cases depend on.

**Use cases** (`usecase/`):
- `AccountService` — creates accounts (validates CPF uniqueness), deposit
- `PixKeyService` — registers keys (enforces 5-key limit), ownership check on delete
- `PaymentService` — `Initiate`: validates funds, creates PENDING tx, publishes Kafka event; `Process` (called by Kafka consumer): debits payer, credits receiver, marks COMPLETED, publishes completion event

### HTTP Adapter (`internal/adapters/in/httpadapter/`)

Package name `httpadapter` (avoids collision with stdlib `net/http`). DTOs defined in `dto.go`. Error mapping from domain errors to HTTP status codes lives in each handler's switch.

Key routes:
- `POST /api/v1/accounts` + `GET /api/v1/accounts/:id` + `GET /api/v1/accounts/:id/balance` + `POST /api/v1/accounts/:id/deposit`
- `POST /api/v1/pix/keys` + `GET /api/v1/pix/keys/:key` + `DELETE /api/v1/pix/keys/:key?account_id=`
- `POST /api/v1/pix/payments` (→ 202 Accepted) + `GET /api/v1/pix/payments/:id`

### Kafka Adapters

**Out** (`internal/adapters/out/kafka/`): `KafkaEventPublisher` publishes to three topics:
- `pix.payment.initiated` — after creating PENDING transaction
- `pix.payment.completed` — after successful processing
- `pix.payment.failed` — after failure

**In** (`internal/adapters/in/kafka/`):
- `PaymentConsumer` — consumes `pix.payment.initiated`, calls `PaymentUseCase.Process()`
- `ResultLogger` — consumes `pix.payment.completed` + `pix.payment.failed`, logs outcome

Both run as goroutines started in `main.go` with graceful shutdown via context cancellation.

### Infrastructure (`internal/infrastructure/`)

- `config/` — reads env vars with Docker-friendly defaults
- `database/` — Postgres connection (pool: 25 open/idle, 5min lifetime)
- `logger/` — global `zap.Logger` + `WithContext`/`FromContext` for request-scoped propagation
- `kafka/` — factory functions for `kafka-go` writer and reader

### Money handling

`Money` is `int64` in centavos — never use `float64` for currency. API accepts/returns `amount_cents` (integer). `Money.String()` formats as `"R$ X.YY"` for display only.

### Swagger

Generated by `swag` from annotations in handler files. Regenerate with `make swag` after modifying any `@Summary`, `@Param`, `@Success`, or `@Router` annotations. The `docs/` directory is generated — do not edit manually.

### Postman

`postman/pix-api.postman_collection.json` + `postman/pix-local.postman_environment.json`. Run folders in order: Setup → Chaves PIX → Pagamentos PIX → Cenários de Erro. Variables (`payer_account_id`, `receiver_account_id`, `pix_key`, `transaction_id`) are auto-populated by test scripts.
